购物车+结算业务逻辑梳理

发表于 2021-03-15 更新于 2021-03-18 分类于 随笔

电商里面都会有类似加入购物车，然后结算的业务。一般是会属于系统里面非常复杂的模块，所以想在这里，从业务逻辑的角度梳理一下。
这条业务完整的逻辑是：选择商品 → 加入购物车 → 订单结算 → 结算成功。
如果走一切正常的逻辑，还是挺简单的，无非是在数据库里面加一条订单结算的记录。但是复杂的地方在于：商品和结算。

商品

电商平台中的商品，会有许多属性，包括：商品名、价格、单位、库存、产地等等。对商品的操作也有很多，包括：修改商品名、修改价格、变更库存、上/下架。商品的属性和操作都会影响到订单结算，所以在订单结算前，需要对待结算单里面的商品，进行多重判断，导致整个业务逻辑非常复杂。

结算

小程序进行微信支付的逻辑如下图。可以简单的理解为：客户端生成支付订单 → 服务器调用微信支付系统 → 微信支付系统生成预付单给服务器 → 客户端获取预付单的信息 → 客户完成支付 → 服务器异步返回支付结果。

其中，比较容易出问题的点在于：客户支付可能成功，也可能失败（比如客户主动取消了支付）；微信支付系统是异步返回结果给服务器，所以不是调用支付完就可以立马得到支付结果。

ThinkPHP 的数据库中间层架构

发表于 2021-03-13 更新于 2021-03-18 分类于 ThinkPHP

一、代码操作数据库的核心

操作数据库的过程核心有两个步骤：

  • 连接数据库
  • 操作数据库

如果只有一种数据库，比如 Mysql，那么直接用面向过程的思路，一步步编写下去即可。但是连接数据库和操作数据库，一定会有的需求：需要连接不同的数据库，并且不同的数据库的操作方法是不一样的。如果按照面向过程的思想，每一个数据库都需要单独写一套连接数据库和操作数据库的方法，会非常麻烦。

这个时候，使用面向对象的封装，就可以很好的解决这个问题。把连接数据库和操作数据库的方法进行封装，不管是连接什么数据库还是这个数据库使用什么的操作方法，都使用同一套方法，把不同数据库的差异性，封装起来。TP 的数据层和数据连接层使用的就是这个方法。

闲谈：计算机和语言

发表于 2020-05-01 更新于 2021-03-18 分类于 随笔

在计算机中，我们经常会碰到精度丢失的问题，比如 100/3=33.33，那么会有 0.01 的精度被丢失掉了。之前以为这种是程序设计的问题，但是后来发现，这个问题的根源，不在于计算机，而在于与计算机紧密相连的数学问题。是数学规定，100/3，就是除不尽。这种问题，可以通过程序设计来规避，比如支付宝的 AA 收款，如果有 100 元，三个人均摊，那么发起 AA 收款的人，需要支付 33.34，另外两个人需要支付 33.33。

另外一种精度问题，是属于计算机的设计缺陷。比如在计算机里，0.1+0.2！=0.3，因为计算机在处理类似等式时，会把他们换成二进制数，再进行相加。这类问题的根源，在于计算机设计时，采用了二进制的计算方法。这种问题，也可以通过程序设计来进行规避。
